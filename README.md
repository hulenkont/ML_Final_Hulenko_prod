# ☔ Двоканальний сервіс прогнозування опадів на основі Open-Meteo API & Machine Learning

## 📝 Опис та призначення проєкту
Цей проєкт є інтерактивним вебзастосунком на базі Streamlit, що реалізує прототип розумного міні-сервісу для автоматизованого збору метеоданих та довгострокового моделювання погоди. 

Головна відмінність цього застосунку — **двоканальна ML-архітектура (Multi-task Pipeline)**. Застосунок не просто передбачає факт дощу за допомогою класифікації, а й паралельно розраховує точний обсяг майбутньої вологи в міліметрах за допомогою регресії, порівнюючи результати власного ШІ з глобальною фізичною моделлю погоди.

---

## 🎯 Мета моделювання
Головне завдання сервісу — побудувати та оптимізувати дві незалежні цільові змінні:
1. **Канал класифікації (Факт опадів):** Бінарний target ($0$ — опадів немає, $1$ — опади є), що розраховується за умовою $precipitation\_sum > 0 \rightarrow 1$.
2. **Канал регресії (Обсяг опадів):** Кількісний target (чисте значення `precipitation_sum` в мм), що передбачає фізичну силу очікуваного шторму чи зливи.

---

## 📡 Використовувані дані та джерела (Data Engineering)
Усі погодні метрики завантажуються в реальному часі через HTTP-запити до безкоштовного європейського метеорологічного сервісу **Open-Meteo**.

* **Історичний архів:** Завантаження щоденних (daily) даних за останні роки через *Historical Weather API* для формування репрезентативного тренувального датасету.
* **Оперативний прогноз:** Завантаження синоптичних карт на 1–14 днів уперед через *Forecast API*.

### Перелік вхідних ознак датасету (14 параметрів):
* `temperature_2m_max`, `temperature_2m_min`, `temperature_2m_mean` — температурні режими дня.
* `apparent_temperature_max`, `apparent_temperature_min` — показники відчутної температури.
* `wind_speed_10m_max`, `wind_gusts_10m_max` — максимальна швидкість та пориви вітру.
* `wind_direction_10m_dominant` — переважний кутовий напрямок вітру.
* `shortwave_radiation_sum` — сумарна сонячна короткохвильова радіація.
* `sunshine_duration`, `daylight_duration` — тривалість сонячного сяйва та світлового дня.
* `et0_fao_evapotranspiration` — рівень потенційного випаровування вологи грунтом.

## ⚙️ Підготовка даних

* **Обробка пропусків:** Лінійна інтерполяція відновлює проміжні значення метеопараметрів, а методи `ffill()` та `bfill()` очищують крайові пропуски, гарантуючи відсутність `NaN`.
* **Feature Engineering (Генерація ознак):** На основі часового індексу автоматично розраховуються циклічні змінні `month` (місяць) та `day_of_year` (день року) для врахування кліматичної сезонності.
* **Видалення витоку даних (Data Leakage):** Показник `precipitation_sum` примусово вилучається з матриці ознак $X$ перед початком навчання, оскільки його наявність призвела б до фіктивного завищення точності.

---

## 🏗️ Структура та архітектура ML-пайплайну

### 1. Валідація часових рядів (Time-Series Split)
Оскільки метеодані мають часову залежність, випадкове перемішування (Shuffle) заборонено. Скрипт реалізує суворе хронологічне розбиття: останні **20% даних** відкладаються як повністю незалежний тестовий набір (`X_test`), а перші **80%** використовуються для крос-валідації через `TimeSeriesSplit(n_splits=5)`.

### 2. Автоматизований відбір ознак (Feature Selection)
Перед передачею даних в алгоритми застосусвується модуль `SelectFromModel`. Він оцінює вагові коефіцієнти ознак «на льоту» та відсіває малозначущі параметри (шум), що упереджує перенавчання (overfitting).

### 3. Оптимізація та стек моделей (GridSearchCV):
* **Канал класифікації:** Оптимізуються `Random Forest Classifier` та лінійна `Logistic Regression` (з масштабуванням через `StandardScaler`). Головна метрика — **F1-Score**.
* **Канал регресії:** Оптимізуються `Random Forest Regressor` та лінійна регресія з регуляризацією `Ridge`. Головні метрики якості — **MAE** та коефіцієнт детермінації **R²**.

## 📊 Оцінка моделей

### Метрики каналу класифікації (Факт дощу):
* `Accuracy` — загальна частка правильно передбачених днів (як сухих, так і дощових).
* `Precision` (Точність) — відсоток реальних дощів серед усіх випадків, де ШІ вказав "🌧️ ТАК" (мінімізація помилкових тривог).
* `Recall` (Повнота) — частка дощових днів, яку модель змогла успішно виявити та передбачити в історії.
* `F1-score` (**Основна метрика**) — гармонійне середнє між Precision та Recall, що об'єктивно оцінює модель в умовах дисбалансу класів.
* `ROC-AUC` — оцінка загальної роздільної здатності алгоритму та стабільності розрахунку ймовірностей.

### Додаткові компоненти оцінки:
* **Confusion Matrix (Матриця помилок):** Інтерактивна теплова карта Plotly, яка наочно розбиває результати тестування на True Positive, True Negative, False Positive та False Negative.
* **Порівняння CV vs Test:** Автоматичне зіставлення метрик крос-валідації та незалежного тесту для контролю стабільності моделей та перевірки на Overfitting.

## 🧭 Послідовність дій та інтерфейс користувача (Workflow)
Програма Streamlit розділена на три логічні взаємопов'язані етапи управління через Sidebar:

### 1️⃣ Блок: Завантаження даних (Load data)
* Користувач гнучко задає локацію (через текстову назву міста або прямі координати) та часовий інтервал.
* Програма завантажує дані та виводить блоки **EDA (Розвідувального аналізу)**: кругову діаграму балансу класів та Box-plot розподілу обсягів дощів по місяцях.

### 2️⃣ Блок: Навчання ML моделі (ML model training)
* Одним натисканням запускається паралельний пошук по решітці для класифікаторів та регресорів.
* Інтерфейс виводить зведені інтерактивні таблиці метрик якості для обох каналів, теплову карту **Confusion Matrix** та графік важливості відібраних ознак (Feature Importance).

### 3️⃣ Блок: Прогноз (Forecast)
* Найважливіший шар інтерфейсу, що формує індивідуальну картку погоди на сьогодні (віджети-картки) та зведену таблицю на 1–14 днів уперед.
* **UI Бенчмаркінг:** Поруч виводяться два показники кількості опадів — розрахований нашою моделлю регресії ШІ та базовий фізичний прогноз від сервера Open-Meteo, а також фінальний графік динаміки температур.
